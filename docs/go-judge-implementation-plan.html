go-judge — Implementation Plan (detailed, phase by phase)

This document turns docs/go-judge-high-level-idea.md into small, codeable, reviewable phases.
Primary learning language: Go (from syntax to backend architecture).

0. How we'll work together

  1. For each phase (or sub-phase) I'll give an outline: Goal, Input, Output, Task list, related Go concepts, Acceptance Criteria, commands to self-verify. May include sample code, but only in chat — never written into your files.
  2. You type the code yourself (no copy-paste) so your hands get used to the syntax.
  3. You report back → I review: correctness, idiomatic or not, logic bugs, over-engineering.
  4. Only move to the next phase once the current one runs and meets the Acceptance Criteria. No phase may depend on an unfinished part of a later phase.
  5. Guiding principle throughout: YAGNI — each phase only builds what that phase needs. Don't write retry/DLQ/Docker sandbox in Phase 1 just because you know it's coming — that's Phase 4/5's job.

Phase 1Monolith Judge (MVP)

Overall goal: Frontend → Backend API → Postgres → run go test directly inside the API process (no gRPC, no queue — matches the MVP scope in the idea doc §8).

1.4 Judge engine: actually run go test, update the verdict

This is the core lesson in os/exec + file handling for the whole project.

Input: 1.3 done.

Output: After creating a submission, the API runs the judge immediately, within the request (synchronous, no async yet — that's Phase 3): write the user's code + test code to a temp directory, run go test, parse the result, update the DB. GET /submissions/{id} now returns a real verdict (PASSED/FAILED/ERROR).

Tasks:

  • [ ] internal/judge.Runner: takes (code string, testCode string) (Result, error)
  • [ ] Create a temp dir: os.MkdirTemp("", "submission_*"), remember defer os.RemoveAll
  • [ ] Write two files: solution.go (package + user code, needs correct package wrapping) and solution_test.go (test code)
  • [ ] Run exec.CommandContext(ctx, "go", "test", "-json", "./...") in that directory (cmd.Dir = tmpDir), use -json for easier parsing than raw text
  • [ ] Parse the output: count passed/failed tests, determine the overall verdict: all pass → PASSED, any fail → FAILED, compile error/timeout → ERROR
  • [ ] Set a timeout with context.WithTimeout (e.g. 10s) — if exceeded, verdict is ERROR with a clear message
  • [ ] Call judge.Runner directly inside the POST /submissions handler (sync), update the submission before responding

Go concepts: os/exec, context.WithTimeout, os.MkdirTemp/defer, handling stdout/stderr, streaming encoding/json decoding (for go test -json) — no goroutines needed here yet (still synchronous), but this is a spot where races show up easily if done carelessly.

Acceptance Criteria:

# Submit a correct solution (idea doc §7 "Expected accepted solution")
curl -X POST localhost:8080/submissions -d '{"problemId":"sum-slice","code":"func Sum(nums []int) int { total := 0; for _, n := range nums { total += n }; return total }"}'
curl localhost:8080/submissions/<id>   # status: PASSED, passed_tests == total_tests

# Submit a wrong solution (starter code from idea doc §7 "return 0")
curl -X POST ... -d '{"code":"func Sum(nums []int) int { return 0 }"}'
curl localhost:8080/submissions/<id>   # status: FAILED

# Submit code that doesn't compile
curl -X POST ... -d '{"code":"func Sum(nums []int) int { retrun 0 }"}'
curl localhost:8080/submissions/<id>   # status: ERROR, output contains the compile error

Security note (remember, do NOT fix in this phase): running go test directly on the host means user code can do anything (read files, hit the network, infinite-loop and burn CPU). This is exactly why Phase 5 (Docker sandbox) exists. Accepting this risk in Phase 1 is fine since it's local, single-user.

  • cmd/runner/main.go: gRPC server implementing RunTests
  • Backend API uses internal/runnerclient (gRPC client) instead of calling judge.Runner directly

  • [ ] cmd/runner/main.go: implement CodeRunnerServiceServer, internally reuse the logic from internal/judge.Runner (move the package, don't rewrite the logic)

Phase 3Add a Message Queue + Worker

Goal: Submissions are processed asynchronously. POST /submissions returns PENDING right away, without waiting for the judge to finish.

Input: Phase 2 done.

Output:

  • docker-compose.yml gains rabbitmq
  • Backend API becomes a producer: after saving the submission as PENDING, publish a message to the judge.submission.created queue and return immediately
  • cmd/worker: a consumer — receives the message, loads the submission, sets it to RUNNING, calls runnerclient.RunTests, updates the verdict, acks the message

Tasks:

  • [ ] internal/queue: Publisher.Publish(ctx, msg) and Consumer.Consume(handler func(msg) error) (wraps amqp091-go)
  • [ ] POST /submissions handler: after the INSERT, publish {submissionId, problemId}, respond right away — remove the direct runner call
  • [ ] cmd/worker/main.go: connect to the queue, consume loop, for each message: load submission → UPDATE status='RUNNING' → call the runner → UPDATE verdict → ch.Ack
  • [ ] Handle bad messages / missing submissions → Nack (no retry/DLQ yet — that's Phase 4)

Go concepts: a goroutine for the consume loop, channels if you need an internal queue, defer conn.Close(), JSON (de)serializing messages, running 3 independent processes (api, worker, runner).

Acceptance Criteria:

docker compose up -d --build

curl -X POST localhost:8080/submissions -d '...'
# → responds immediately (< 100ms), status PENDING

# Poll a few seconds later
curl localhost:8080/submissions/<id>
# → RUNNING then PASSED/FAILED (worker processes it asynchronously, worker container logs show the message being consumed)

docker compose logs worker   # shows "received job", "verdict: PASSED"

Phase 4Reliability

Goal: The system doesn't lose submissions when the worker crashes, doesn't double-process, and shuts down cleanly.

Input: Phase 3 done, queue already observed working under normal conditions.

Output:

  • Retry when the runner fails transiently (network/timeout) — up to N times with backoff
  • Dead-letter queue for messages that fail more than N times
  • Idempotent processing: reprocessing the same submissionId doesn't produce a wrong result (check status before overwriting, or use the retry_count field from idea doc §5)
  • Graceful shutdown: worker receives SIGTERM, stops accepting new jobs, waits for in-flight jobs to finish (with a timeout) before exiting
  • Overall timeout per submission (if the runner hangs longer than X seconds → ERROR, the worker never hangs forever)

Tasks:

  • [ ] Add a retry_count column to the submissions table (new migration)
  • [ ] Worker: when the runner call fails for infrastructure reasons (not a code-test failure) → increment retry_count, requeue (Nack with requeue=true) if retry_count < max, otherwise route to the DLQ
  • [ ] Configure a DLQ in RabbitMQ (separate exchange + queue, x-dead-letter-exchange on the main queue)
  • [ ] Idempotency check: at the start of the consumer, if the submission is already PASSED/FAILED/ERROR → ack immediately, don't reprocess
  • [ ] signal.NotifyContext(ctx, os.Interrupt, syscall.SIGTERM) in cmd/worker/main.go, use this context throughout the consume loop + sync.WaitGroup to wait for in-flight jobs

Go concepts: serious context cancellation propagation, signal.NotifyContext, sync.WaitGroup, backoff (hand-written, simple: time.Sleep(time.Duration(retryCount) * time.Second) — no dedicated backoff lib yet, YAGNI), thinking in terms of "at-least-once delivery".

Acceptance Criteria:

# Test graceful shutdown: submit a long-running job, then kill the worker
docker compose kill -s SIGTERM worker
# → log "shutting down, waiting for N in-flight jobs", the job still finishes with a verdict before the worker exits

# Test retry: stop code-runner, submit a job
docker compose stop code-runner
curl -X POST ...
docker compose logs worker   # shows "retry 1/3", "retry 2/3"...
docker compose start code-runner
# → the job eventually gets PASSED once the runner comes back (within the retry limit)

# Test idempotency: manually publish 2 messages with the same submissionId to the queue → final verdict is consistent, no error

Phase 5Docker Sandbox for code execution

Goal: User code no longer runs directly on the Code Runner's host — it runs in a temporary container with resource limits and no network.

Input: Phase 4 done.

Output: When code-runner receives RunTests, instead of exec.Command("go", "test", ...) directly, it will:

  1. Write the code to a temp directory (unchanged)
  2. Run docker run --rm --network none --memory=128m --cpus=0.5 --read-only -v <tmpdir>:/app -w /app golang:1.2x go test ./... with a timeout
  3. Clean up the container + temp dir afterward (even on error/timeout — defer)

Tasks:

  • [ ] Mount the Docker socket into the code-runner container (or run code-runner outside Docker Compose's network for simplicity — decide based on how complex Docker-in-Docker gets, weigh carefully before doing it)
  • [ ] internal/judge.Runner (or the equivalent package under cmd/runner): change from local go test to docker run ... with the resource-limit flags above
  • [ ] exec.CommandContext(ctx, "docker", "run", ...) with a timeout context — if exceeded, ctx.Err() → kill the container (use --rm + cancel the context and call cmd.Process.Kill())
  • [ ] Ensure cleanup: container self-removes with --rm, temp dir via defer os.RemoveAll

Go concepts: still os/exec + context, but now thinking about "process isolation" — resource limits enforced at the OS/container level, not in Go code.

Acceptance Criteria:

# Code that intentionally infinite-loops
curl -X POST ... -d '{"code":"func Sum(nums []int) int { for {} }"}'
curl localhost:8080/submissions/<id>   # ERROR: timeout, doesn't hang the whole code-runner (submit another job right after — still works fine)

# Code that tries to read system files / hit the network
curl -X POST ... -d '{"code":"... os.ReadFile(\"/etc/passwd\") ..."}'
# → fails due to --read-only / --network none, no host data leaks

docker ps   # after the submission finishes, no leftover containers for that submission

Phase 6Realtime Result

Goal: Replace crude polling with a better frontend experience.

Input: Phase 3+ already has polling working (baseline).

Output (do sequentially, each step is its own milestone, none mandatory beyond step 1):

  1. Polling (already exists from Phase 1.5) — keep as-is, this is the baseline that always works even if the steps below break.
  2. SSE: endpoint GET /submissions/{id}/stream — server holds the connection open, pushes an event when status changes (subscribe via Postgres LISTEN/NOTIFY, or simpler: the worker calls back into an in-memory pub/sub in the API after updating — start with the simpler option).
  3. WebSocket (optional, only if you want to try it) — not required for this project's learning goals.

Tasks (SSE):

  • [ ] Handler GET /submissions/{id}/stream: set SSE headers (Content-Type: text/event-stream), loop sending events, use ctx.Done() to exit when the client disconnects
  • [ ] Notification mechanism: simplest first — have the API itself poll the DB every 500ms inside the SSE handler and diff the status (do NOT need Postgres LISTEN/NOTIFY for the first version — YAGNI, only upgrade if polling-inside-SSE is actually a bottleneck)
  • [ ] Frontend: use EventSource instead of setInterval polling on the /submissions/:id page

Go concepts: http.Flusher, streaming responses, tracking client disconnects via context (r.Context().Done()), the deliberate trade-off between "simpler" and "architecturally correct" — here we choose simple on purpose.

Acceptance Criteria:

curl -N localhost:8080/submissions/<id>/stream
# → multiple lines like "event: status\ndata: {...}" streaming in real time, no need to call curl again
